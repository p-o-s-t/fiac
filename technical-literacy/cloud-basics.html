Cloud Service Models

Cloud computing transforms physical server resources into dynamic, logical services hosted over the internet, categorized into three core layers:

  • IaaS (Infrastructure as a Service): Virtualized compute, storage, and networking. The customer installs and manages the OS and applications (e.g., AWS EC2, Azure VMs).
  • PaaS (Platform as a Service): Pre-configured execution environments. The customer only manages application code and data (e.g., Google App Engine, AWS Elastic Beanstalk).
  • SaaS (Software as a Service): Completed web-hosted applications. The customer only manages settings and access permissions (e.g., Microsoft 365, Salesforce).

Deployment Models & Architecture

Organizations customize their cloud environment based on compliance, risk, and scalability requirements:

  • Public Cloud: Shared infrastructure owned by third-party hyperscalers (AWS, Azure, Google Cloud) and accessed over the public internet.
  • Private Cloud: Infrastructure dedicated entirely to a single organization, hosting applications on-premise or in private datacenters.
  • Hybrid & Multi-Cloud: Integrating on-premise resources with public clouds, or orchestrating microservices across multiple cloud vendors using Kubernetes.

The Shared Responsibility Model

A critical cloud security concept is the dividing line of security duties between the provider and the customer:

  • Provider Duties (Security OF the Cloud): Securing physical datacenters, server host machines, hypervisors, and global core network routing.
  • Customer Duties (Security IN the Cloud): Securing databases, configuring firewalls (Security Groups), patching guest operating systems, and managing identity and access controls (IAM).
  • Risk: The majority of cloud breaches target customer responsibilities, specifically IAM misconfigurations.

Cloud-Specific Attack Vectors

Threat actors exploit the unique architectural features of cloud environments to gain initial access and move laterally:

  • Exposed Storage Buckets: Misconfiguring AWS S3 buckets or Azure Blobs to allow anonymous public reading, leading to massive data leaks.
  • Credential Leaks: Accidentally uploading hardcoded cloud API access keys to public repositories (such as GitHub).
  • IAM Role Assumption: Exploiting over-privileged server profiles to assume administrator roles and move laterally to other cloud projects.

Cloud Threat Intelligence & Auditing

To investigate cloud intrusions, threat analysts pivot from analyzing host-level hard drives to parsing massive log streams:

  • API Audit Logs: Analyzing services like AWS CloudTrail, Google Cloud Audit Logs, and Azure Activity Logs to track who made what API call.
  • Network Logs: Querying VPC Flow Logs to trace network connections, data volumes, and protocols moving across virtual network segments.
  • Anomalous Activity: Flagging suspicious API behaviors, such as logins from unexpected countries or bulk data exfiltration from storage buckets.
